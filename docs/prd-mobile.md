# PomodoroX 移动端适配 PRD

> **文档版本**: v1.0 | **更新日期**: 2026-05-03 | **状态**: 草案
>
> 本文档为 PomodoroX 从桌面端扩展到手机浏览器的增量 PRD，仅描述本次变更范围。

---

## 1. 产品目标

### 目标一：手机浏览器全功能可用

PomodoroX 五个核心模块（计时器、任务、反思、统计、设置）在手机浏览器上完整可用，所有交互均适配触摸操作。用户从桌面迁移到手机不会丢失任何功能入口。

- 不新增功能模块，仅对现有功能做触控适配
- 关键路径（开始/暂停计时、创建/完成任务、写反思）在手机端操作步骤不增加
- 计时器的呼吸动画、动态背景、音效等沉浸体验在手机端保持效果

### 目标二：数据零丢失，跨端无缝一致

手机端数据通过 `localStorage` + GitHub 同步双保障，刷新不丢失，与桌面端共享同一套 GitHub 数据仓库。

- 手机端使用 `localStorage` 作为本地存储（已有兜底实现），跳过 SQLite（仅 Tauri 环境可用）
- GitHub 同步（Octokit REST API）在手机浏览器直接工作，与桌面端共用 `github.ts`
- 手机端创建/修改的数据同步到 GitHub 后，桌面端刷新即可看见
- 手机端首次使用时，从 GitHub 拉取已有数据完成初始化

### 目标三：响应式 UI，适配主流手机屏幕

UI 在 320px ~ 414px 宽度（主流手机屏幕）上完整可用，布局自然折叠，不出现横向滚动或元素挤压。

- 底部 Tab 导航保持 5 项，图标 + 极短标签
- 计时器环形 SVG 自适应缩放
- 任务列表/看板/统计图表等复杂视图纵向滚动
- 支持 safe-area-inset 适配刘海屏/挖孔屏

## 2. 用户故事

### US-01：在路上快速开始一个番茄钟

> 作为知识工作者，我在通勤路上打开手机浏览器访问 PomodoroX，点击"开始专注"即可启动一个 25 分钟番茄钟。计时器全屏沉浸显示，倒计时清晰可见。我无需任何额外配置。

**验收标准**：
- 手机浏览器打开 `http://<局域网IP>:1420` 即显示计时器页面
- 点击主按钮（大字 START）立即开始倒计时
- 进度环和数字在手机屏幕上清晰可读
- 计时结束后有通知提醒（通过浏览器 Notification API）

### US-02：手机端创建任务并关联番茄钟

> 作为用户，我在手机上新建一个"写周报"任务，回到计时器页面选择该任务开始专注。番茄钟完成后，任务的实际番茄数自动 +1。

**验收标准**：
- 任务页面的"新建任务"按钮可点击，弹窗在手机上全屏显示
- 输入标题、选择优先级、设置标签均触控友好
- 计时器页面的任务选择器在手机端可正常打开和选择

### US-03：碎片时间写反思日志

> 作为用户，我在午休时打开手机，在反思页面选择今天日期，写下"上午效率不错"并选择一个开心表情。内容自动保存，不丢失。

**验收标准**：
- Markdown 编辑区在手机上可正常输入和预览
- 心情选择器触控友好（点击而非悬停）
- 内容在离开页面或刷新后不丢失（localStorage 自动保存）

### US-04：手机端查看自律统计

> 作为用户，我在睡前用手机查看今天的专注统计：完成了几个番茄钟、专注时长多久、心情趋势如何。图表在手机上正常显示。

**验收标准**：
- 统计卡片的四宫格布局在手机端自适应为两列或纵向堆叠
- 柱状图/折线图/甜甜圈图在手机屏幕完整显示
- 日期范围选择器在手机上触控可用

### US-05：手机端设置 GitHub 同步

> 作为新用户，我在手机上首次使用时填入 GitHub Token、Owner 和 Repo，点击"测试连接"验证通过。数据同步到云端，后续在桌面端登录同一仓库即可看到所有数据。

**验收标准**：
- 设置页面的 GitHub 配置区域在手机上完整显示
- Token 输入框支持密码模式
- 测试连接按钮反馈正确
- 配置保存后，数据同步按钮正常工作

## 3. 竞品分析

选择 7 款主流番茄钟或专注类工具，重点分析其多端方案。

| 产品 | 桌面端 | 移动端方案 | PWA/离线 | 数据同步 | 核心发现 |
|------|--------|-----------|----------|---------|---------|
| **Forest** | Win/Mac 原生 | iOS/Android 原生 | 否 | 各端独立（有云同步付费） | 双端原生体验最好，但桌面与移动是独立产品 |
| **潮汐** | Mac 原生 | iOS/Android 原生 | 否 | 账号云同步 | 沉浸感最强，但无 Web 端 |
| **Pomofocus** | Web (PWA) | Web (PWA) | ✅ | 无（仅有导出） | 与 PomodoroX 最相似，纯 Web 方案，无原生依赖 |
| **Be Focused** | Mac 原生 | iOS 原生 | 否 | iCloud 同步 | Mac + iPhone 生态封闭 |
| **TickTick** | Win/Mac 原生 + Web | iOS/Android 原生 | Web 版支持 | 账号云同步 | 功能最全，但番茄钟只是其中一小部分 |
| **Todoist** | Win/Mac 原生 + Web | iOS/Android 原生 | Web 版支持 | 账号云同步 | 任务管理为主，番茄钟为附加功能 |
| **Focus To-Do** | Win/Mac 原生 | iOS/Android 原生 | 否 | 账号云同步 | 番茄钟 + 任务管理集成度好，但有广告 |

### 竞品启示

1. **Pomofocus 是 PomodoroX 最直接对标的竞品** — 纯 Web、PWA、番茄钟 + 简单任务。PomodoroX 的差异化在于：GitHub 数据自主控制 + 反思日志 + 深度统计。
2. **纯 Web 方案（Pomofocus 模型）** 对 PomodoroX 天然适配 — 当前技术栈（Vue 3 + Vite）构建的页面已可在浏览器运行，Tauri 仅作为桌面壳。
3. **数据同步是关键痛点** — 多数竞品依赖自有账号体系。PomodoroX 的 GitHub 同步方案无需注册，天然跨端，是独特优势。
4. **PWA 加装是低成本方案** — 无需开发原生 App，Service Worker + Manifest 即可提供"添加到主屏幕"体验。

## 4. 技术规范和需求池

### 4.1 架构变更

当前 PomodoroX 在浏览器模式（`pnpm dev`）下已可运行，数据层使用 MemoryStore 回退。移动端适配需做以下变更：

```
┌──────────────────────────────┐
│          手机浏览器            │
│  ┌────────────────────────┐  │
│  │  Vue 3 前端 (当前不变)   │  │
│  │  - 所有视图组件          │  │
│  │  - Pinia Stores          │  │
│  │  - Composables           │  │
│  └────────┬───────────────┘  │
│           ▼                   │
│  ┌────────────────────────┐  │
│  │  数据层适配              │  │
│  │  database.ts            │  │
│  │  → 非 Tauri 环境自动    │  │
│  │    走 MemoryStore+      │  │
│  │    localStorage 持久化   │  │
│  │  github.ts (不变)        │  │
│  └────────────────────────┘  │
└──────────────────────────────┘
```

### 4.2 关键技术决策

| 决策项 | 方案 | 理由 |
|--------|------|------|
| 本地存储 | localStorage（已有 MemoryStore 兜底） | SQLite 仅 Tauri 可用，localStorage 在手机浏览器完全可用 |
| 路由模式 | `createWebHashHistory()` | `createWebHistory()` 需要服务端配合，Hash 路由在静态部署时无问题 |
| 响应式方案 | Tailwind CSS 断点 + 组件内媒体查询 | 已有 Tailwind，补充移动端断点即可 |
| 触控适配 | 保持原生 `<button>` + `<input>`，不引入 touch 库 | 浏览器原生 touch 支持已足够，避免额外依赖 |
| 通知 | 浏览器 Notification API（已有 `useNotification.ts` 回退） | 已在 Tauri 环境实现回退，手机浏览器可直接使用 |
| PWA | 增加 manifest.json + Service Worker | 用户在手机端获得"类原生 App"体验，成本极低 |
| 键盘快捷键 | 手机端自动降级为不显示快捷键提示 | 手机无物理键盘，快捷键 UI 元素应隐藏 |

## 5. UI 设计稿说明

> 以下为响应式布局的视觉描述，供 UI 设计和前端开发参考。不包含高保真设计图。

**桌面端（> 1024px）**：
```

├────────────────────────────────┤
│                                │
│         主内容区域               │

├────────────────────────────────┤
│  ⏱️  📋  💭  📊  ⚙️            │  ← 底部导航 64px

```

**手机端（< 640px）**：
```

├────────────────┤
```

### 5.2 底部导航

- **桌面**：图标 + 2 字中文标签（专注、任务、反思、统计、设置）
- **手机**：图标 + 1 字极短标签（专、任、反、统、设）
- **选中态**：图标上移 2px + 底部圆点指示器
- **安全区域**：底部 padding 增加 `env(safe-area-inset-bottom, 0px)`

### 5.3 计时器页面（TimerView）

**桌面（环形半径 120px）**：
```
     ╭─────────────╮
     │   ⏱️ 25:00   │
     │  /‾‾‾‾‾‾‾\  │
     │ ( ◯ 环形进度 ) │
     │  \_________/  │
     │  [开始]       │
     │  今日: 0 个   │
     ╰─────────────╯
```

**手机（环形半径约 80~90px，视屏幕宽度动态计算）**：
```
  ┌──────────────┐
  │   ⏱️ 25:00    │← 字号适当缩小
  │  ┌────────┐  │
  │  │  ◯ 环形  │  │← 自适应半径
  │  │  [开始]  │  │← 触控友好 ≥48px
  │  └────────┘  │
  │  今日: 0 个  │
  └──────────────┘
```

- 进度环 SVG 的 `viewBox` 不变，通过 CSS `width/height` 百分比 + `max-width` 限制实现自适应
- 主按钮最小 48px 高度
- 模式切换（专注/短休/长休/自由）在手机上改为全宽横向滚动选择卡

## 6. 待确认问题

以下问题需要在开发前与团队确认：

### Q1：是否优先支持 PWA？

PWA 可以显著提升手机端体验（添加到主屏幕、离线访问），但需要额外开发 Service Worker。建议作为 P1 在本次迭代中完成。

**建议**：PWA 作为 P1 纳入，与核心功能同步开发。

### Q2：是否兼容 QQ/微信内置浏览器？

国内用户可能在 QQ/微信中直接打开链接。但这些 WebView 的兼容性问题较多（不支持 Service Worker、部分 ES 语法不支持）。

**建议**：P0 以主流手机浏览器为目标（Chrome/Safari/Firefox 移动版），P2 再处理 QQ/微信兼容。

### Q3：手机端是否需要"横屏锁定"？

番茄计时器在横屏下布局可能不合理（两侧大量空白），竖屏是更自然的专注姿态。

**建议**：添加 CSS 提示，鼓励用户竖屏使用，但不强制锁定。

### Q4：手机首次启动数据初始化流程？

用户首次在手机端使用时，如果 GitHub 仓库已有数据，是否需要自动从 GitHub 拉取？

**建议**：手机端 `onMounted` 时检测 `localStorage` 是否为空，若为空且有配置 GitHub 连接，则自动触发一次拉取。若未配置 GitHub，使用空数据。

### Q5：手机端计时器后台运行时如何保持？

手机浏览器在后台或锁屏时可能被系统挂起，导致计时器不准。当前基于 `Date.now()` 时间戳的方案已有一定的后台恢复能力（恢复时重新计算剩余时间），但通知可能延迟或不触发。

**建议**：使用 `document.visibilitychange` 事件在页面重新可见时立即刷新计时状态。注册 Service Worker 后可使用 Push Notification 改善通知体验（P2）。

### Q6：Toast 和 Modal 组件在手机端的行为？

当前 Modal 宽度固定 `420px`，在手机上需要全屏显示。Toast 位置也需要调整。

**建议**：Modal 在手机上改为全屏底部弹出（drawer 风格），Toast 从底部导航上方弹出。

### Q7：是否需要处理手机端的返回手势？

Android 系统返回键（或左滑手势）在 SPA 中可能触发浏览器后退，与 App 内导航冲突。

**建议**：检测 `popstate` 事件，在手机端显示退出确认提示，防止误触丢失数据。

### Q8：Vite build 后的产物如何部署到手机可访问？

当前 `pnpm build` 构建的是 Tauri 专用产物。手机端需要构建纯前端产物（`--mode mobile` 或修改配置）。

**建议**：新增 `pnpm build:web` 命令，构建纯前端产物到 `dist-web/` 目录，可直接部署到任何静态服务器或通过 `npx serve dist-web` 局域网访问。
