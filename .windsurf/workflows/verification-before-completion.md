# Verification Before Completion — 验证驱动完成

## Overview

宣称工作完成而不验证，是欺骗而非高效。

**核心原则：** 证据先于声明，永远。

**违反字面规则即违反精神规则。**

## 铁律

```
没有最新验证证据，不得宣称完成。
```

如果在本轮对话中没有运行过验证命令，就不能说它通过了。

## Gate 函数

```
在宣称状态或表达满意之前：

1. 识别：什么命令能证明这个声称？
2. 运行：执行完整的命令（新鲜的、完整的）
3. 读取：完整输出，检查退出码，统计失败数
4. 验证：输出确认声称吗？
   - 如果否：用实际状态 + 证据说明
   - 如果是：用证据说明声称
5. 然后：才能做出声称

跳过任何一步 = 撒谎，不是验证。
```

## 常见失败模式

| 声称 | 需要 | 不足够 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 个失败 | 之前跑过、"应该会通过" |
| Linter 干净 | Linter 输出：0 个错误 | 部分检查、外推 |
| 构建成功 | 构建命令：exit 0 | Linter 通过、日志看起来不错 |
| Bug 修复 | 测试原始症状：通过 | 代码改了、假定修复了 |
| 回归测试有效 | 红绿循环已验证 | 测试通过一次 |
| 代理完成 | VCS diff 显示变更 | 代理报告"成功" |
| 需求满足 | 逐项核对 checklist | 测试通过 |

## Red Flags — STOP

如果你发现自己在想或说：
- "应该可以了"、"可能行"、"看起来对了"
- 在验证之前表达满意（"很好！"、"完美！"、"完成了！"等）
- 即将 commit/push/PR 但不验证
- 相信代理的成功报告
- 依赖部分验证
- 想着"就这一次"
- 累了想赶紧结束
- **任何暗示成功但没有运行验证的措辞**

## 反理性化

| 借口 | 现实 |
|------|------|
| "应该可以了" | 运行验证命令。 |
| "我很确信" | 确信 ≠ 证据。 |
| "就这一次" | 没有例外。 |
| "Linter 通过了" | Linter ≠ 编译器。 |
| "代理说成功了" | 独立验证。 |
| "我累了" | 累 ≠ 借口。 |
| "部分检查就够了" | 部分证明不了什么。 |
| "换了说法所以规则不适用" | 精神重于字面。 |

## 关键模式

**测试：**
```
✅ [运行测试命令] [看到: 34/34 pass] "所有测试通过"
❌ "应该可以通过了" / "看起来正确"
```

**回归测试（TDD 红绿）：**
```
✅ 编写 → 运行(通过) → 撤销修复 → 运行(必须失败) → 恢复 → 运行(通过)
❌ "我已经写了回归测试"（没有红绿验证）
```

**构建：**
```
✅ [运行构建] [看到: exit 0] "构建通过"
❌ "Linter 通过了"（linter 不检查编译）
```

**需求：**
```
✅ 重新读取计划 → 创建 checklist → 逐项验证 → 报告差距或完成
❌ "测试通过了，阶段完成"
```

**代理委托：**
```
✅ 代理报告成功 → 检查 VCS diff → 验证变更 → 报告实际状态
❌ 相信代理报告
```

## 何时应用

**始终在以下情况之前：**
- 任何关于成功/完成的表述
- 任何满意表达
- 关于工作状态的任何积极陈述
- 提交、PR 创建、任务完成
- 移动到下一个任务
- 委托给代理

**规则适用于：**
- 确切的措辞
- 释义和同义词
- 暗示成功
- 任何暗示完成/正确的沟通

## 底线

**验证没有捷径。**

运行命令。读取输出。**然后**声称结果。

这是不可协商的。
