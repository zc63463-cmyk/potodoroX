# 项目工作流编排器

根据用户的具体意图，自动选择合适的开发流程。

## 路由逻辑

检查用户的输入，匹配以下意图：

| 用户意图 | 关键词 | 执行流程 |
|---------|--------|---------|
| 新功能开发 | "新功能" / "feature" / "开发" / "实现" | → 流程 A |
| Bug 修复 | "bug" / "修复" / "fix" / "问题" | → 流程 B |
| 重构优化 | "重构" / "refactor" / "优化" / "架构" | → 流程 C |
| 发布准备 | "发布" / "release" / "部署" / "上线" | → 流程 D |

如果意图不明确，使用 **grill-me** 追问澄清。

## 流程 A：新功能开发

```
用户输入 → 设计探索 → 生成 PRD → 拆解 Issues → 编写实施计划 → TDD 开发 → 架构检查 → 配置 hooks
```

### 步骤

0. **【入口确认点】** ✋ 展示流程计划，等待用户确认
   - 展示将要执行的步骤概要：
     ```
     📋 流程 A 执行计划：
     1. 设计探索（调用 brainstorming）
     2. 生成 PRD 文档（调用 to-prd）
     3. 拆解开发任务（调用 to-issues）
     4. 编写实施计划（调用 writing-plans）
     5. TDD 循环开发（调用 tdd）
     6. [可选] 架构深度检查（调用 improve-architecture）
     7. [可选] 配置提交检查（调用 setup-pre-commit）
     ```
   - 用户可选操作：
     - **"开始"** → 继续执行
     - **"跳过 X"** → 跳过某一步骤
     - **"只做 X"** → 只执行指定步骤
     - **"退出"** → 终止流程

1. **设计探索**
   - 调用 `brainstorming` skill
   - 通过 Socratic 式问题澄清用户需求
   - 提出 2-3 种方案并推荐
   - 逐段呈现设计，逐段获得批准
   - 输出设计文档到 `docs/specs/YYYY-MM-DD-<topic>-design.md`
   - ✋ **确认点**：展示设计文档摘要
   - 您可以说：
     - **"继续"** → 确认设计方向，进入 PRD 生成
     - **"调整 XX"** → 修改设计中的某部分
     - **"跳过"** → 跳过设计探索，直接进入 PRD
     - **"退出"** → 终止流程

2. **生成 PRD 文档**
   - 调用 `to-prd` skill
   - 从对话上下文提取需求
   - 输出 PRD 并发布到 GitHub Issues
   - **PRD 输出格式示例**：
     ```markdown
     # [功能名称] PRD
     ## 背景与目标
     - 为什么要做这个功能？
     ## 用户故事
     - As a [角色], I want [功能], so that [价值]
     ## 功能需求
     - [需求1]: 描述...
     - [需求2]: 描述...
     ## 非功能需求
     - 性能: ...
     - 安全: ...
     ## 验收标准
     - [标准1]: ...
     ```
   - ✋ **确认点**：展示 PRD 大纲
   - 您可以说：
     - **"继续"** → 确认 PRD 方向，进入下一步
     - **"调整 XX"** → 修改 PRD 的某部分
     - **"重新生成"** → 重新生成 PRD
     - **"退出"** → 终止流程

3. **拆解开发任务**
   - 调用 `to-issues` skill
   - 将 PRD 拆解为可独立执行的 Issues
   - 按依赖关系排序
   - **Issue 输出格式示例**：
     ```markdown
     ## Issue #{N}: [标题]
     - 优先级: P0/P1/P2
     - 依赖: #{N-1}, #{N-2}
     - 验收标准:
       1. [标准1]
       2. [标准2]
     - 预估工作量: [S/M/L]
     ```
   - ✋ **确认点**：展示 Issue 列表
   - 您可以说：
     - **"继续"** → 确认拆分，开始编写实施计划（步骤4）
     - **"调整顺序"** → 修改 Issue 的依赖关系
     - **"合并 XX"** → 合并某些 Issue
     - **"退出"** → 终止流程

4. **编写实施计划**
   - 调用 `writing-plans` skill
   - 将 Issue 拆解为 bite-sized 的详细实施步骤（每个步骤 2-5 分钟）
   - 每个步骤包含：完整代码、精确文件路径、验证命令
   - **输出格式示例**：
     ```markdown
     ### Task N: [组件名]

     **Files:**
     - Create: `src/auth/middleware.ts`
     - Test: `tests/auth/middleware.test.ts`

     - [ ] **Step 1: Write the failing test**
       ```typescript
       test('rejects unauthenticated requests', () => { ... });
       ```
     - [ ] **Step 2: Run test to verify it fails**
       Expected: FAIL
     - [ ] **Step 3: Write minimal implementation**
       ```typescript
       function authMiddleware(req, res, next) { ... }
       ```
     - [ ] **Step 4: Run test to verify it passes**
       Expected: PASS
     - [ ] **Step 5: Commit**
     ```
   - ✋ **确认点**：展示实施计划摘要
   - 您可以说：
     - **"继续"** → 确认计划，开始 TDD 开发
     - **"调整 XX"** → 修改某 Task 的具体内容
     - **"补充 Task"** → 增加遗漏的实施步骤
     - **"跳过"** → 跳过编写实施计划，直接进入 TDD
     - **"退出"** → 终止流程

5. **TDD 循环开发**
   - 调用 `tdd` skill
   - 对每个 Issue 依次执行编写实施计划(writing-plans) → 红绿重构循环
   - 每完成一个 Issue 更新进度
   - ✋ **确认点**：Issue 开发完成
   - 您可以说：
     - **"继续"** → 开始下一个 Issue
     - **"暂停"** → 暂停开发，保存进度
     - **"重做"** → 重新开发当前 Issue
     - **"退出"** → 终止流程

6. **架构深度检查**（可选）
   - 如果用户说 "检查架构" / "优化架构"
   - 调用 `improve-architecture` skill
   - 识别浅模块，提出深化建议
   - ✋ **确认点**：展示架构建议
   - 您可以说：
     - **"继续"** → 确认架构建议，进入下一步
     - **"调整"** → 修改架构建议
     - **"跳过"** → 跳过架构检查
     - **"退出"** → 终止流程

7. **配置提交检查**
   - 调用 `setup-pre-commit` skill
   - 确保代码质量门禁生效
   - ✋ **确认点**：展示 pre-commit 配置
   - 您可以说：
     - **"继续"** → 确认配置，完成流程
     - **"添加 XX"** → 添加其他检查项
     - **"跳过"** → 跳过 pre-commit 配置
     - **"退出"** → 终止流程

## 流程 B：Bug 修复

```
问题描述 → 诊断根因 → 编写回归测试 → 修复 → 验证
```

### 步骤

0. **【入口确认点】** ✋ 展示流程计划，等待用户确认
   - 展示将要执行的步骤概要：
     ```
     🐛 流程 B 执行计划：
     1. 纪律性诊断（调用 diagnose）
     2. [可选] 追问验证（调用 grill-me，复杂问题时）
     3. TDD 修复循环（调用 tdd）
     4. [可选] 文档同步（调用 grill-with-docs，有 CONTEXT.md 时）
     ```
   - 用户可选操作：
     - **"开始"** → 继续执行
     - **"跳过诊断"** → 直接到步骤2或3
     - **"只要修复"** → 只执行 TDD 修复循环
     - **"退出"** → 终止流程

1. **纪律性诊断**
   - 调用 `diagnose` skill
   - 复现问题 → 最小复现 → 提出假设 → 仪器化验证
   - 输出根因分析报告
   - ✋ **确认点**：展示根因分析
   - 您可以说：
     - **"继续"** → 确认根因，进入修复方案
     - **"补充 XX"** → 补充其他线索
     - **"重新诊断"** → 重新运行诊断
     - **"退出"** → 终止流程

2. **追问验证**（复杂问题时）
   - 调用 `grill-me` skill
   - 验证修复方案的正确性
   - 考虑边界情况和副作用
   - ✋ **确认点**：展示修复方案
   - 您可以说：
     - **"继续"** → 确认方案，开始 TDD 修复
     - **"调整 XX"** → 修改修复方案
     - **"补充边界"** → 补充边界情况处理
     - **"退出"** → 终止流程

3. **TDD 修复循环**
   - 先写失败的回归测试（红）
   - 修复代码使测试通过（绿）
   - 重构优化（重构）
   - ✋ **确认点**：测试通过
   - 您可以说：
     - **"继续"** → 继续重构优化
     - **"暂停"** → 暂停，保存进度
     - **"完成"** → 跳过重构，完成修复
     - **"退出"** → 终止流程

4. **文档同步**（如果项目有 CONTEXT.md）
   - 调用 `grill-with-docs` skill
   - 更新文档中的相关决策记录
   - ✋ **确认点**：展示文档更新内容
   - 您可以说：
     - **"继续"** → 确认更新，完成流程
     - **"调整 XX"** → 修改文档内容
     - **"跳过"** → 跳过文档同步
     - **"退出"** → 终止流程

## 流程 C：重构优化

```
当前架构分析 → 识别浅模块 → 深化设计 → 逐步重构 → 验证
```

### 步骤

0. **【入口确认点】** ✋ 展示流程计划，等待用户确认
   - 展示将要执行的步骤概要：
     ```
     🔧 流程 C 执行计划：
     1. 架构深度分析（调用 improve-architecture）
     2. 架构决策追问（调用 grill-with-docs）
     3. TDD 安全重构（调用 tdd）
     4. 更新项目文档（CONTEXT.md + ADRs）
     ```
   - 用户可选操作：
     - **"开始"** → 继续执行
     - **"只分析不重构"** → 只执行步骤1-2
     - **"直接重构"** → 跳过步骤1-2，直接开始重构
     - **"退出"** → 终止流程

1. **架构深度分析**
   - 调用 `improve-architecture` skill
   - 读取 `CONTEXT.md` 中的领域语言
   - 识别浅模块（通过"删除测试"思维实验）
   - ✋ **确认点**：展示浅模块列表
   - 您可以说：
     - **"继续"** → 确认选择，开始重构
     - **"调整"** → 修改模块选择
     - **"全部重构"** → 重构所有浅模块
     - **"退出"** → 终止流程

2. **架构决策追问**
   - 调用 `grill-with-docs` skill
   - 追问深化方案，同步更新 ADRs
   - 确保新设计与项目领域语言一致
   - ✋ **确认点**：展示架构决策
   - 您可以说：
     - **"继续"** → 确认决策，进入下一步
     - **"调整 XX"** → 修改架构决策
     - **"重新分析"** → 重新分析架构
     - **"退出"** → 终止流程

3. **TDD 安全重构**
   - 调用 `tdd` skill
   - 先完善测试覆盖（红）
   - 执行重构（绿 → 重构）
   - 确保每次重构后测试全通过
   - ✋ **确认点**：模块重构完成
   - 您可以说：
     - **"继续"** → 继续下一个模块
     - **"暂停"** → 暂停重构，保存进度
     - **"重做"** → 重新重构当前模块
     - **"退出"** → 终止流程

4. **更新项目文档**
   - 更新 `CONTEXT.md` 中的模块描述
   - 新增或更新 `docs/adr/` 中的架构决策记录
   - ✋ **确认点**：展示文档更新摘要
   - 您可以说：
     - **"继续"** → 确认更新，完成流程
     - **"调整 XX"** → 修改文档内容
     - **"跳过"** → 跳过文档同步
     - **"退出"** → 终止流程

## 流程 D：发布准备

```
代码冻结 → 测试验收 → 架构审查 → 版本号 → Changelog → 发布
```

### 步骤

0. **【入口确认点】** ✋ 展示流程计划，等待用户确认
   - 展示将要执行的步骤概要：
     ```
     🚀 流程 D 执行计划：
     1. 全量测试验收（运行测试套件）
     2. 架构最终审查（调用 improve-architecture）
     3. 生成发布文档（Release Notes + Changelog）
     4. [可选] 配置发布 hooks（pre-commit + CI/CD）
     ```
   - 用户可选操作：
     - **"开始"** → 继续执行
     - **"只测试不发布"** → 只执行步骤1-2
     - **"只要文档"** → 只执行步骤3
     - **"配置 CI/CD"** → 执行步骤4（会触发二次确认）
     - **"退出"** → 终止流程

1. **全量测试验收**
   - 运行完整测试套件
   - 调用 `tdd` skill 确保所有测试通过
   - 检查测试覆盖率
   - ✋ **确认点**：展示测试结果
   - 您可以说：
     - **"继续"** → 确认结果，进入下一步
     - **"补充测试"** → 增加测试用例
     - **"重新测试"** → 重新运行测试
     - **"退出"** → 终止流程

2. **架构最终审查**
   - 调用 `improve-architecture` skill
   - 快速检查是否有浅模块引入
   - 验证模块接口设计合理性
   - ✋ **确认点**：展示架构审查结果
   - 您可以说：
     - **"继续"** → 确认结果，进入下一步
     - **"处理 XX"** → 处理指定架构问题
     - **"重新审查"** → 重新审查架构
     - **"退出"** → 终止流程

3. **生成发布文档**
   - 调用 `to-prd` skill（复用，生成 Release Notes）
   - 整理 Changelog
   - 更新版本号
   - ✋ **确认点**：展示 Release Notes 草稿
   - 您可以说：
     - **"继续"** → 确认文档，进入下一步
     - **"调整 XX"** → 修改发布文档
     - **"跳过"** → 跳过发布配置
     - **"退出"** → 终止流程

4. **配置发布 hooks**

   ⚠️ **本步骤包含破坏性操作，将修改项目配置和 Git 历史**

   - **检查 pre-commit 配置**
     - 运行 `git config --get core.hooksPath` 检查 hooks 路径
     - 如果未配置，调用 `setup-pre-commit` skill
     - 验证 pre-commit 在提交时能正常触发
   - **配置 CI/CD 自动发布**（可选，需二次确认）
     - 如果用户说 "配置 CI/CD" / "自动发布"
     - ✋ **二次确认点**：明确告知用户将要执行的操作：
       ```
       ⚠️ 即将执行以下破坏性操作：
       1. 创建/覆盖 CI/CD 配置文件
       2. 后续推送将自动触发构建和发布
       3. 可能需要修改 GitHub/GitLab 仓库设置
       
       确认要继续吗？输入"确认配置 CI/CD"继续，输入其他内容取消
       ```
     - 根据用户确认结果决定是否继续
     - 根据项目类型选择方案：
       - **GitHub Actions**：创建 `.github/workflows/release.yml`
       - **Jenkins**：创建 `Jenkinsfile`
       - **GitLab CI**：创建 `.gitlab-ci.yml`
     - 发布流程包含：测试 → 构建 → 打 tag → 发布 Release
   - ✋ **确认点**：展示完整的发布配置
   - 您可以说：
     - **"确认发布"** → 执行发布
     - **"调整 XX"** → 修改发布配置
     - **"测试后再发"** → 返回测试步骤
     - **"退出"** → 终止流程

## 流程串联与组合调用

### 串联决策规则

当用户的需求涉及多个流程时，按以下规则决定执行顺序：

1. **依赖关系优先**: 如果流程 X 的输出是流程 Y 的输入，先执行 X
   - 例: "重构后再开发新功能" → 先执行流程 C，再执行流程 A
2. **风险隔离**: 修复类流程优先于发布类流程
   - 例: "修复bug并发布" → 先执行流程 B，再执行流程 D
3. **可选流程询问**: 串联时询问用户是否执行可选步骤
   - 例: "开发功能" → 询问"是否需要同时配置 pre-commit?"

### 串联示例 1: 重构 + 新功能开发

```
用户: "我要开发用户登录功能，顺便把现在的 session 管理重构一下"

路由结果:
  → 流程 C (重构优化) + 流程 A (新功能开发) 串联

执行顺序:
  1. 流程 C: 先重构 session 管理（为登录功能铺路）
     - 步骤0: 入口确认点（展示重构计划）
     - 步骤1-4: 执行完整重构流程
     - 确认点: 重构完成，询问是否继续
  2. 流程 A: 开发登录功能
     - 步骤0: 入口确认点（展示开发计划）
     - 步骤1-5: 设计探索(brainstorming) → 生成PRD → 拆解Issues → 编写实施计划(writing-plans) → TDD开发
  3. [可选] 流程 B: 为登录功能编写安全测试
     - 询问用户: "是否需要为登录功能编写安全测试？"

串联确认点:
  - 每个流程结束后，询问是否继续执行下一个流程
  - 用户可以说:
    - "继续" → 执行下一个流程
    - "暂停" → 保存进度，暂停执行
    - "跳过 XX" → 跳过某个流程
    - "退出" → 终止所有流程
```

### 串联示例 2: Bug修复 + 发布准备

```
用户: "修复登录闪退的bug，然后发布补丁版本"

路由结果:
  → 流程 B (Bug 修复) + 流程 D (发布准备) 串联

执行顺序:
  1. 流程 B: 修复登录闪退 bug
     - 步骤0: 入口确认点（展示诊断计划）
     - 步骤1-3: 诊断根因 → TDD修复 → 验证
  2. 流程 D: 准备补丁版本发布
     - 步骤0: 入口确认点（展示发布计划）
     - 步骤1-3: 全量测试 → 架构审查 → 生成Release Notes
     - 步骤4: [可选] 配置CI/CD（需二次确认）

串联确认点:
  - Bug修复完成后，询问: "Bug已修复，是否继续发布准备？"
  - 发布文档生成后，询问: "Release Notes已生成，是否执行发布？"
```

### 串联示例 3: 复杂多流程组合

```
用户: "我的项目有点乱，需要重构架构，修复几个bug，然后发布新版本"

路由结果:
  → 流程 C (重构) + 流程 B (修复) + 流程 D (发布) 串联

执行顺序:
  1. 流程 C: 架构重构（最优先，为后续工作铺路）
  2. 流程 B: 修复bug（重构可能引入新bug，需验证）
  3. 流程 D: 发布新版本（最后执行）

风险提醒:
  - 在流程 C 执行前，提醒用户: "大规模重构可能影响现有功能，建议先创建备份分支"
  - 在流程 D 执行前，确认: "确保所有bug已修复，测试已全部通过"
```

### 流程中断与恢复

如果串联执行过程中用户选择"暂停"或"退出"：

```
暂停处理:
  1. 保存当前进度到 `.workbuddy/checkpoints/process_{id}.json`
  2. 记录:
     - 当前流程: 流程 C
     - 当前步骤: 步骤3 (TDD安全重构)
     - 已完成: 流程 C 步骤1-2
     - 待执行: 流程 C 步骤3-4, 流程 A, 流程 B
  3. 提示用户: "进度已保存，输入'继续'可恢复执行"

恢复处理:
  1. 用户说 "继续" 或 "恢复上次进度"
  2. 读取最近的检查点文件
  3. 展示进度摘要，询问: "是否从流程 C 步骤3 继续执行？"
  4. 用户确认后，从该步骤继续执行
```

### 跨流程数据共享

当多个流程串联执行时，前一个流程的输出自动作为后一个流程的输入：

```
示例: 流程 A → 流程 C 串联

  - PRD 文档 → 传递给流程 C 作为重构需求背景

流程 C 读取:
  - 从流程 A 接收的 PRD 文档
  - 从流程 A 接收的 Issue 列表
  - 从项目读取现有的测试覆盖报告
```

## 错误处理与边界条件

### 流程执行中的错误处理

#### 1. Skill 调用失败

```
错误场景: 调用 to-prd 时失败（如 gh 未认证、网络超时、权限不足）

处理步骤:
  1. 捕获错误信息，分类为以下类型:
     - 环境问题: gh 未安装、Python 版本不兼容、依赖缺失
     - 权限问题: GitHub 未认证、无仓库访问权限
     - 网络问题: API 超时、DNS 失败、代理配置错误
     - 参数问题: 缺少必需参数、参数格式错误
  
  2. 调用 diagnose skill 进行诊断:
     - 输入: 错误信息 + 当前上下文
     - 输出: 根因分析 + 修复建议
  
  3. 根据诊断结果执行 Fallback:
     - 环境问题 → 提示用户修复环境，提供具体命令
     - 权限问题 → 引导用户完成认证流程
     - 网络问题 → 重试 3 次，失败后提示用户检查网络
     - 参数问题 → 追问用户提供必要的参数
  
  4. Fallback 失败后的最后手段:
     - 输出错误信息 + 诊断报告到本地文件
     - 提示用户: "自动执行失败，您可以手动执行该步骤"
     - 提供手动执行的具体命令或步骤
```

#### 2. 用户中断处理

```
中断场景: 用户在执行过程中说 "暂停" / "退出" / "停止"

处理步骤:
  1. 立即保存当前进度:
     - 保存到: `.workbuddy/checkpoints/workflow_{timestamp}.json`
     - 内容:
       {
         "flow": "A",
         "current_step": 2,
         "completed_steps": [0, 1],
         "context": { "prd": "...", "issues": [...] },
         "timestamp": "2026-05-03T00:45:00"
       }
  
  2. 根据用户选择执行不同动作:
     - "暂停" → 保存进度，提示: "进度已保存到 .workbuddy/checkpoints/，输入'继续'可恢复"
     - "退出" → 保存进度，询问: "是否确认退出？进度已保存，可随时恢复"
     - "停止" → 同 "退出"
  
  3. 恢复执行:
     - 用户说 "继续" / "恢复" → 读取最新检查点，展示进度摘要
     - 用户确认后，从断点继续执行
```

#### 3. 依赖 Skills 缺失

```
缺失场景: 调用 to-prd 时，但 to-prd skill 未安装

处理步骤:
  1. 检测缺失:
     - 尝试调用 skill，捕获 "skill not found" 错误
     - 检查 ~/.workbuddy/skills/ 目录，确认 skill 是否存在
  
  2. 提示用户:
     - 输出: "⚠️ 缺少依赖 skill: to-prd"
     - 提供安装命令: "`skills install to-prd` 或从 marketplace 安装"
     - 询问: "是否跳过此步骤，或手动执行？"
  
  3. Fallback 方案:
     - 如果用户选择 "跳过" → 记录跳过原因，继续后续步骤
     - 如果用户选择 "手动执行" → 提供详细的手动执行步骤
     - 如果用户选择 "安装" → 等待安装完成后重试
```

### 边界条件处理

#### 1. 空输入或意图不明确

```
场景: 用户只说 "开发" / "修复" / "重构"，没有具体描述

处理步骤:
  1. 触发 grill-me skill 进行追问:
     - 追问模板:
       ```
       🔍 需要更多信息才能继续：
       
       1. 具体要开发/修复/重构什么功能？
       2. 有相关的 Issue 或文档吗？
       3. 有没有特别的要求或约束？
       
       请提供更多信息，或者直接说 "跳过追问" 使用默认流程。
       ```
  
  2. 用户说 "跳过追问" → 使用默认流程，基于最小假设执行
  3. 用户说 "取消" → 终止流程，返回主菜单
```

#### 2. 项目环境异常

```
场景: 项目缺少必要文件（如 package.json、requirements.txt、CONTEXT.md）

处理步骤:
  1. 检测缺失文件:
     - 流程 A/B/C/D 开始前，检查项目根目录
     - 列出缺失的文件清单
  
  2. 根据缺失文件执行不同动作:
     - 缺少 CONTEXT.md → 提示: "建议先创建 CONTEXT.md，或跳过文档同步步骤"
     - 缺少配置文件 → 提示: "项目配置不完整，可能影响 Skill 调用"
     - 缺少测试文件 → 提示: "未检测到测试框架，TDD 步骤可能需要先配置"
  
  3. 提供修复建议:
     - 输出: "建议先执行以下操作："
     - 列出具体的修复步骤（如 `npm init`、`pip install pytest`）
     - 询问: "是否先修复环境问题，还是继续（可能失败）？"
```

#### 3. 并发流程冲突

```
场景: 用户在一个会话中同时触发多个流程（如先说 "开发功能"，又说 "修复bug"）

处理步骤:
  1. 检测冲突:
     - 如果当前有流程正在执行，提示:
       "⚠️ 当前有流程正在执行（流程 A，步骤2），是否：
       - 暂停当前流程，开始新流程
       - 继续执行当前流程
       - 串联两个流程（先完成当前，再执行新流程）"
  
  2. 根据用户选择执行:
     - 暂停 → 保存当前进度，开始新流程
     - 继续 → 忽略新请求，继续当前流程
     - 串联 → 将新流程加入队列，当前流程结束后自动执行
  
  3. 流程队列管理:
     - 保存到: `.workbuddy/checkpoints/workflow_queue.json`
     - 用户可以说: "查看队列" / "取消队列" / "优先执行 XX"
```

#### 4. 大项目性能问题

```
场景: 项目代码量巨大（如 10000+ 文件），Skill 调用超时或卡顿

处理步骤:
  1. 检测项目规模:
     - 执行前，运行: `find . -type f | wc -l` 估算文件数量
     - 如果 > 5000 文件，提示:
       "⚠️ 项目规模较大（N 个文件），可能需要较长时间处理"
  
  2. 提供优化建议:
     - 建议: "是否使用 .gitignore 排除无关文件？"
     - 建议: "是否只分析核心模块（如 src/、lib/）？"
     - 建议: "是否分批处理（每次处理一个模块）？"
  
  3. 执行降级方案:
     - 如果用户确认继续 → 增加超时时间（从 120s 到 600s）
     - 如果仍然超时 → 输出部分结果，提示用户手动补充
```

### Git 操作安全检查

#### 1. 破坏性 Git 操作前确认

```
操作清单: 以下操作会修改 Git 历史或远程仓库，必须二次确认：
  - `git push --force`
  - `git reset --hard`
  - `git rebase`
  - `git branch -D`
  - 修改 `.gitignore` 或 `.git/` 目录

确认模板:
  ```
  ⚠️ 即将执行破坏性 Git 操作：
  
  操作: git push --force
  
  影响:
  1. 会覆盖远程仓库的提交历史
  2. 可能影响其他协作者的工作
  3. 数据丢失风险：无法恢复被覆盖的提交
  
  确认要继续吗？
  - 输入 "确认执行" → 继续执行
  - 输入其他内容 → 取消操作，使用安全模式（git push --force-with-lease）
  ```
```

#### 2. CI/CD 配置覆盖检查

```
场景: 配置 CI/CD 时，发现已存在配置文件（如 .github/workflows/ 已有文件）

处理步骤:
  1. 检测冲突:
     - 检查目标路径是否已存在文件
     - 如果存在，读取现有配置内容
  
  2. 提示用户:
     - 输出: "⚠️ 检测到已存在的 CI/CD 配置："
     - 展示: 现有配置文件名 + 内容摘要
     - 询问: "要覆盖还是合并？"
  
  3. 根据用户选择执行:
     - "覆盖" → 备份现有配置（添加 .bak 后缀），写入新配置
     - "合并" → 尝试智能合并（保留现有 job，添加新 job）
     - "跳过" → 跳过 CI/CD 配置步骤
```
